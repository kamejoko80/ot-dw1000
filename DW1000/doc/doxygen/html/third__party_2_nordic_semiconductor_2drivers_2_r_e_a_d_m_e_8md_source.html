1 # nRF 802.15.4 radio driver.
2 
3 This driver implements only __non-beacon mode__ of operation.
4 It supports following __features__:
5  * reception of unicast and broadcast frames (with filtering),
6  * automatic sending ACK frames,
7  * setting pending bit in ACK frame according to pending data for given
8  destination,
9  * transmission of unicast and broadcast frames,
10  * automatic CCA procedure before transmission,
11  * automatic receiving ACK frames,
12  * low power mode (sleep),
13  * energy detection,
14  * promiscuous mode.
15 
16 ## Implementation details
17 
18 The driver is a FSM. From API perspective it has 4 states. Most of those states
19 contains sub-states in implementation.
20 
21 ### FSM description
22 
23 ```
24  receive() transmit()
25  --------> -------->
26  Sleep Receive Transmit
27  <-------- | /|<--------
28  sleep() | | receive() / transmitted() / busy_channel()
29  | |
30  energy_detection() | | energy_detected()
31  \|/ |
32  Energy detection
33 ```

35 #### Transitions
36 
37 The driver is initialized in the Sleep state. The higher layer should call
38 the receive() function to make the driver enter the Receive state and start
39 radio operations.
40 
41 In basic applications radio should be most time in a Receive state. In this
42 state the radio receives incoming frames. Changing to any other state should be
43 performed from Receive state.
44 
45 When a frame is received in Receive state the driver notifies the higher layer
46 by calling received() function. This function is called after reception of a
47 broadcast frame or after sending an ACK to a unicast frame.
48 In the promiscuous mode the higher layer is notified about all of the received
49 frames. Even if the frame was not destined to the receiving node.
50 
51 To transmit a frame the higher layer should call the transmit() function. If
52 channel is busy the driver goes back to the Receive state and notifies the
53 higher layer by calling the busy_channel() function. If a broadcast frame was
54 transmitted the driver goes back to the Receive state and notifies the higher
55 layer by calling the transmitted() function. If a unicast frame was transmitted
56 and an ACK was received the driver goes back to the Receive state and notifies
57 the higher layer by calling the transmitted() function. If a unicast frame was
58 transmitted and there was no expected ACK received the higher layer shall call
59 the receive() function after the ACK timeout to make the driver go back to the
60 Receive state.
61 
62 To perform an Energy Detection procedure the higher layer should call the
63 energy_detection() function. When the procedure is completed the driver goes
64 automatically back to the Receive state and notifies the higher layer with the
65 energy_detected() function.

67 #### States
68 
69 ##### Sleep
70 In this state the radio is in low power mode. It cannot transmit or receive any
71 frame.
72 
73 ##### Receive
74 In this state the radio receives 802.15.4 frames. It filters out frames with
75 invalid CRC, length, type, destination address.
76 If the driver receives unicast frame destined to the receiving node it
77 automatically transmits an ACK frame. According to 802.15.4 standard, an ACK
78 frame should be delayed aTurnaroundTime (192 uS) after reception of the ACKed
79 frame. To perform this delay the driver uses the TIFS timer in the radio
80 peripheral. This timer requires 3 shorts to work correctly:
81 1. END -> DISABLE
82 2. DISABLE -> TXEN
83 3. READY -> START
84 The driver has limited time after receiving of a frame to decide if an ACK
85 should be transmitted. If ACK should not be transmitted the driver must abort
86 sending ACK by disabling those shorts and triggering DISABLE task.
87 To use this limited time most effective the driver uses the Bit Counter feature
88 of the radio peripheral to get notification when the Frame Control field is
89 received and when the destination address is received. Those fields are used to
90 filter the frame before whole frame is received.
91 If all 3 shorts used to send ACK automatically are enabled the radio peripheral
92 sends ACK frames in loop. To prevent this during debugging process there are
93 only 2 shorts enabled when waiting for frame (1. and 2.) and 2 other shorts are
94 enabled in DISABLED event handler (1. and 3.). The first short is still enabled
95 to automatically disable transmitter after transmission of the ACK frame.
96 
97 ##### Transmit
98 In this state the radio performs the CCA procedure. If channel is free the radio
99 transmits requested frame. If an ACK was requested in the transmitted frame
100 the driver automatically receives the ACK frame in this state.
101 To prevent the TXIDLE peripheral state the driver uses 2 shorts in Transmit
102 state:
103 1. READY -> START
104 2. END -> DISABLE
105 Those shorts automatically start transmission of the frame when transmitter is
106 ready and disable transmitter when the frame was transmitted.
107 
108 ##### Energy detection
109 In this state the radio performs the Energy Detection procedure. During this
110 procedure the radio is busy and cannot change state to any other. The end of
111 this procedure is notified to the higher layer by a function call.
112 
113 ### Mutex and critical sections.
114 
115 State transitions in the FSM can be requested simultaneously by the higher layer
116 and the IRQ handler. To prevent race conditions in the driver there is a mutex.
117 The mutex is unlocked only in the *Receive* state (*WaitingRxFrame* substate).
118 If there is requested state transition, the procedure shall lock the mutex
119 before state is changed. If mutex cannot be locked, another procedure has locked
120 it and is going to change the state.
121 The mutex is unlocked when the driver enters *Receive* state.
122 
123 A race condition could also occur during handle of a requests from the higher
124 layer. Even if the receiver is stopped (TASK STOP) the END or DISABLED event can
125 be raised for a few uS after triggering the task. To prevent interrupt of the
126 higher layer request handler by IRQ handler, the higher layer request handlers
127 are performend in critical sections. The critical sections are implemented as
128 software interrupt requests with priority equal to the RADIO IRQ.
